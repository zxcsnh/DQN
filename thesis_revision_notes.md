## 一、需要优先修正的项目事实

当前项目实际实现如下，论文中涉及实现细节时应以这些内容为准：

- 支持环境：`taxi`、`mountaincar`、`dino`，分别对应 `Taxi-v4`、`MountainCar-v0`、自定义 `TrexEnv-v0`。
- 支持算法：普通 `DQN` 与 `PER-DQN`。`PERDQNAgent` 继承 `DQNAgent`，主要区别是将普通回放池替换为 `PrioritizedReplayBuffer`。
- Q 网络结构不是 `Linear-ReLU-Linear-ReLU-Linear`，而是：

```text
Linear -> LayerNorm -> SiLU -> Linear -> LayerNorm -> SiLU -> Linear
```

- 损失函数不是均方误差，而是 `SmoothL1Loss`，即 Huber loss；PER-DQN 中再乘以重要性采样权重。
- 目标网络更新不是统一硬更新：Taxi 和 MountainCar 默认硬更新；Dino 默认软更新，`tau=0.001`。
- 当前默认配置未启用 Double DQN，代码只是保留了可选实现。
- Dino 状态不是 15 维，而是 23 维。
- Dino 奖励不是“每步 0.1、过障碍 +2、碰撞 -10、冗余跳跃 -0.02”，而是“每步 +0.01、碰撞 -1.0、过障碍额外奖励为 0、冗余跳跃惩罚为 0”；过障碍数仍作为 `obstacles_cleared` 指标统计。
- Dino 当前训练参数不是 5000 回合、batch 256，而是 3000 回合、batch 128。
- PER 参数 `alpha` 当前为 0.5，不是 0.45；`beta_start=0.4`，`beta_anneal_steps=180000`，`priority_epsilon=1e-4`。

## 二、摘要与绪论中的结果表述需要收敛

### 1. 摘要

原文中“结果表明，PER DQN 在各类任务中均能有效完成策略学习，在奖励稀疏和长期决策环境中展现出良好的收敛效率和策略稳定性”过于绝对。当前项目已有实验结果显示不同环境、不同随机种子下 DQN 与 PER-DQN 的优势并不完全一致，尤其 Dino 场景中 PER-DQN 并非稳定优于 DQN。因此建议改为更谨慎的表达：

> 实验从回报、成功率、训练步数、策略稳定性及环境特定指标等多个角度比较 DQN 与 PER-DQN 的表现。实验结果表明，优先经验回放能够改变关键样本的利用方式，并在部分任务和随机种子下提升学习效率；但其增益受环境奖励结构、状态特征和超参数设置影响，不同任务中的表现存在差异。

摘要中“超参数优化”建议改为“超参数设置与环境适配”，因为当前项目没有系统化调参搜索。

### 2. 1.3 主要研究内容

原文已经提前写出“结果表明，PER DQN 能够在各类任务中高效完成策略学习”。如果第五章尚未正式写完，建议改为：

> 实验从平均回报、成功率、训练步数、策略稳定性以及环境特定指标等方面评估 DQN 与 PER-DQN 的性能差异，并分析优先经验回放机制在不同任务中的适用性。

这样更符合当前“结果部分待补写”的状态，也避免与后续真实实验结果冲突。

## 三、第三章模型设计部分修改

### 1. 3.5 Q 网络结构设计

原文写的是 ReLU 与加权均方误差。应改为：

> 本项目所用 Q 网络采用轻量级多层感知机结构。网络输入维度由环境状态表示决定，输出维度等于动作空间大小。具体结构为：输入层后接两层全连接隐藏层，每个隐藏层之后依次使用 LayerNorm 和 SiLU 激活函数，最后通过全连接输出层得到各离散动作的 Q 值。与简单 ReLU 结构相比，LayerNorm 能够缓解不同状态特征尺度差异对训练的影响，SiLU 激活函数则提供平滑的非线性表达能力。

损失函数部分建议改为：

> 训练时，当前 Q 网络输出样本状态下执行动作对应的估计值，目标网络计算目标 Q 值。普通 DQN 使用 Huber loss 计算 TD 目标与当前估计之间的差异；PER-DQN 在此基础上引入重要性采样权重，对每个样本的损失进行加权，从而在提高高优先级样本采样概率的同时修正非均匀采样带来的偏差。

### 2. 3.6 目标网络更新机制设计

原文写“本文采用硬更新方式同步目标网络参数”，与 Dino 配置不一致。建议改为：

> 本项目同时支持硬更新和软更新两种目标网络同步方式。Taxi 与 MountainCar 环境默认采用硬更新，即每隔固定训练步数将当前 Q 网络参数复制到目标网络；Chrome Dino 环境默认采用软更新，即在每次训练更新后按照系数 `tau=0.001` 对目标网络参数进行平滑更新。软更新能够降低目标值突变，有利于持续决策环境中的训练稳定性。

### 3. 3.7 算法流程

“目标网络同步”步骤应从“按设定频率复制参数”改为：

> 根据当前环境配置选择目标网络更新方式：若使用硬更新，则每隔指定训练步数复制当前网络参数；若使用软更新，则按比例将当前网络参数融合到目标网络参数中。

### 2. 4.2 环境介绍总述

### 3. 4.2.3 Chrome Dino 状态空间

原文 15 维状态需要整体替换为 23 维状态。建议写为：

> 本环境采用人工特征方式构造状态输入，而非直接使用游戏图像。当前状态空间为 23 维连续向量，由 7 个基础状态特征和最近两个障碍物的特征组成。基础状态特征包括：小恐龙离地高度、竖直速度、是否正在跳跃、是否正在下蹲、当前游戏速度、是否可以跳跃、最近两个障碍物之间的水平间距。每个障碍物使用 8 个特征描述，包括障碍物是否存在、障碍物类型、与小恐龙的水平距离、按当前速度估计的碰撞时间、障碍物宽度、障碍物高度、障碍物中心纵坐标、障碍物中心与小恐龙中心的相对纵坐标。因此总维度为 `7 + 8 * 2 = 23`。

建议状态表改为：

| 特征类别 | 维度 | 含义 |
| --- | ---: | --- |
| 基础状态 | 7 | 离地高度、竖直速度、跳跃状态、下蹲状态、游戏速度、可跳跃标志、最近两个障碍物间距 |
| 第一个障碍物 | 8 | 是否存在、类型、距离、预计碰撞时间、宽度、高度、中心 y 坐标、相对 y 坐标 |
| 第二个障碍物 | 8 | 同上 |
| 合计 | 23 | 归一化后输入 Q 网络 |

其中“距离”应表述为“障碍物左边缘与小恐龙右边缘之间的水平距离”，不是“障碍物左边缘与障碍右边缘之间的水平距离”。

### 4. 4.2.3 Chrome Dino 奖励函数

原奖励函数与代码不一致，建议替换为：

> 当前实现中的 Chrome Dino 奖励函数以生存时间为主要优化目标。每个时间步给予 `+0.01` 的基础生存奖励；若发生碰撞，则额外给予 `-1.0` 的惩罚并终止当前回合。成功越过障碍物会更新 `obstacles_cleared` 指标，但当前版本不再给予额外正奖励；空中重复跳跃也不再给予额外惩罚。因此，训练回报主要反映智能体的存活时长，而 `obstacles_cleared` 用于补充衡量避障能力。

公式可写为：

```text
r_t = 0.01 - 1.0 * I_collision
```

其中 `I_collision` 表示当前时间步是否发生碰撞。若论文希望强调避障能力，建议不要只看 `total_reward`，还应报告 `obstacles_cleared`、`score`、`steps` 和成功率。当前 Dino 成功判定为 `obstacles_cleared >= 20`。
